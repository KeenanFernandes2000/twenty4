# M9.5 — Moments (experimental)

> Spec phase: post-MVP **experiment** (sits after M9, before the M10+ full-product track) · Depends on: M6 (capture), M7 (montage), M9 (day model) + a **minimal push** capability built here · Branch commit: one commit on `rebuild/v2` (gated, opt-in)
> **Status: EXPERIMENTAL.** The point of this milestone is to *evaluate* whether prompted, synchronized capture is worth keeping. It must be shippable behind an opt-in and fully measurable, and removable if the data says no.

## 1. Goal
An opt-in **"Moments mode"**: instead of capturing whenever, the app sends **5–6 synchronized prompts per day** (timed per timezone). When a moment fires, every opted-in user in that timezone is notified at the same time and has **~2 minutes** to capture a photo/video of what they're doing right now. The day's moments **are** the montage source; the existing review → publish flow (M7) then takes over. Success = enough signal to decide whether prompted capture beats free-form capture on engagement/authenticity.

## 2. Scope
- **In scope:**
  - **Per-timezone moment scheduler:** a daily job picks **6** random times (default; server-tunable) inside a waking-hours window (default 09:00–22:00 local) per timezone. One shared schedule per tz/day so everyone opted-in is prompted together.
  - **Minimal push** (bundled here per decision): `expo-notifications` + push-token registration + a `dispatch-moment` job that pushes to all opted-in users in a tz at each scheduled time. (Full notification suite still lands at M11, reusing this.)
  - **2-min capture window**, late capture allowed but **flagged `late`**; a fully missed moment leaves a gap (shorter montage).
  - **Purely-moments montage sourcing:** in Moments mode, the montage pulls that day's moment-clips **in prompt order**; the M7 review screen still lets the user drop a clip, regenerate, and pick theme/music before publishing.
  - **Per-user opt-in toggle** ("Try Moments mode") stored in `notification_prefs`.
  - **Today UI for moments:** the day's 6 slots showing captured / on-time / late / missed.
  - **Analytics for evaluation:** prompt-sent, captured (on-time/late), missed, montage-from-moments published — enough to compare opted-in vs normal-flow behaviour.
- **Explicitly out of scope (deferred unless the experiment validates):**
  - Streaks / gamification, BeReal-style "RealMojis", dual front+back capture, retake limits.
  - Changing the flow for **non-opted-in** users (the MVP free-form flow is untouched for them).
  - The full M11 notification catalogue (reminders, friend-activity pushes) — only moment-prompt push is built here.

## 3. Tasks (ordered checklist)
- [ ] **Opt-in flag:** extend `notification_prefs` with `moments_enabled` (default false); `PATCH /users/me` accepts it; expose in settings + an onboarding nudge.
- [ ] **Push capability (minimal):** register Expo push tokens (`expo-notifications`), store per device; a `dispatch-moment` BullMQ job sends to opted-in users in a tz. **(jobId must not contain `:` — use `-`.)**
- [ ] **Scheduler:** a daily per-timezone job that generates `moment_schedule` rows (6 random times in the window, `ordinal` 1..6). Use `Intl`-based tz handling consistent with M9's 4am day-window logic; idempotent per (tz, day_bucket).
- [ ] **Dispatch:** at each `scheduled_at`, enqueue `dispatch-moment` for that tz; record send.
- [ ] **Capture binding:** reuse the M4 `POST /media` init/complete, adding `moment_id`; server stamps `captured_at`, computes `is_late` against the moment's `scheduled_at` + window.
- [ ] **Montage sourcing:** in Moments mode, M7 builds the EDL from the day's `daily_media_item`s where `moment_id` is set, ordered by the moment `ordinal`; gaps simply mean fewer segments.
- [ ] **Mobile — Moments surface:** today screen shows the 6 slots (locked until fired, 2-min countdown when active, captured/late/missed states); deep-link from the push opens straight into the capture window.
- [ ] **Analytics events** wired (see §2) behind the §12 firewall (no content, allow-listed dims).
- [ ] **Config:** moment count + window + capture-window length are server config (tune during the test).
- [ ] **Tests** (see §7) green against the live stack.

## 4. Data model & migrations
- **`moment_schedule`** (new) — `id` · `timezone` · `day_bucket` (date) · `scheduled_at` (timestamptz) · `ordinal` (1..N) · unique (`timezone`,`day_bucket`,`ordinal`).
- **`daily_media_item`** (extend) — add `moment_id uuid?` (FK → `moment_schedule`) · `is_late boolean default false` · optional `capture_latency_ms int?`.
- **`user.notification_prefs`** (jsonb, no new column) — add `moments_enabled` key.
- **Push tokens** — a small `push_token` table (`user_id`, `expo_token`, `platform`, `tz`, `updated_at`) or fold into an existing device row; needed for dispatch.
- Migration: next sequential Drizzle migration; enums (if any new state) in `enums.ts`.

## 5. API endpoints
| Method | Path | Auth | Purpose |
|---|---|---|---|
| PATCH | `/users/me` | session | Toggle `moments_enabled` (+ other prefs). |
| POST | `/push/register` | session | Register/refresh this device's Expo push token + timezone. |
| GET | `/moments/today` | session | The user's day: the 6 scheduled moments, their states (pending/active/captured/late/missed). |
| POST | `/media` (extended) | session | Init upload with optional `moment_id` (reuses M4); `/media/:id/complete` stamps `is_late`. |
| GET | `/admin/moments/config` · PATCH | admin | Read/tune count, window, capture-window (experiment knobs). |

## 6. Mobile (Expo Go, Android)
- **Settings:** "Try Moments mode" toggle (+ an onboarding nudge once available).
- **Today (Moments view):** 6 slots; active moment shows a 2-min countdown and a one-tap capture; states for captured / late / missed.
- **Push deep-link:** tapping the moment notification opens the capture window directly.
- Capture reuses the M6 camera + upload subsystem; the montage/review/publish reuses M7.
- ⚠️ **Expo Go push caveat** — see §11: remote push may require a dev build.

## 7. Tests (live-stack)
- **Scheduler:** generates exactly N moments per (tz, day) inside the window, distinct ordinals, idempotent on re-run.
- **Window/late:** capture inside 2 min → `is_late=false`; after → `is_late=true`; assert `captured_at`/latency.
- **Montage sourcing:** with K of 6 captured, the EDL has K segments in ordinal order; a missed moment yields a gap (no crash).
- **Opt-in gating:** non-opted-in users get **no** prompt and the normal free-form flow; opted-in users get the moment flow.
- **Dispatch:** opted-in users in a tz are enqueued at `scheduled_at`; opted-out excluded.
- **Analytics:** prompt/captured/late/missed/published events fire with content-free, allow-listed payloads.

## 8. Acceptance criteria
- An opted-in account on a real Android device **receives a moment prompt**, captures within the 2-min window, and the clip is bound to that moment (`is_late=false`).
- The day's captured moments build a montage **in prompt order**; missed moments just shorten it; the user can review/edit and publish via the existing flow.
- Non-opted-in users are unaffected (normal MVP flow).
- The experiment is **evaluable:** opt-in rate, per-moment capture completion (on-time vs late vs missed), and montage-from-moments publish rate are all queryable to compare against the free-form flow.
- **Android device check:** opt in on the device → trigger a moment (a test/admin-forced fire is fine) → get the push → capture in-window → see it become a montage clip → publish to a group.

## 9. Dependencies & prerequisites
- **M6** capture + upload subsystem; **M7** montage pipeline + review/publish; **M9** day-window + media model.
- **Push:** `expo-notifications`, Expo push tokens, a dispatcher — **built minimally here**.
- **Services:** Postgres, Redis/BullMQ (scheduler + dispatch), per M0 infra.
- **Env:** moment count / window / capture-window config; Expo push credentials.

## 10. Learnings to apply (from PHASE1_WORK_RECAP.md)
- **Expo Go limits native features** — remote push may not work in Expo Go (recent SDKs); this is the same class of issue as the upload native-module gap. Decide the dev-client-vs-fallback path up front (see §11), don't discover it on-device.
- **Per-timezone scheduling** mirrors the M9 4am day-window: use `Intl`, persist resolved times on rows, don't recompute at read.
- **BullMQ jobId must not contain `:`** — the scheduler/dispatch jobs (`moment-<tz>-<day>-<ordinal>`) use `-`.
- **Reuse the proven upload subsystem** (platform-split + streaming fallback) for moment captures — don't fork it.
- **Analytics firewall** — moment events carry no content and only allow-listed dimensions.

## 11. Open decisions / flags
- **Expo Go push vs dev-client** — remote push notifications were dropped from Expo Go in recent SDKs. **Options:** (a) pull the **dev-client (M13)** forward for this experiment, or (b) use **on-device scheduled local notifications** seeded from `moment_schedule` for a rough first test (weaker reliability/sync). **Default: start with the local-notification fallback to keep it in Expo Go; move to a dev build if the signal looks promising.**
- **Success threshold** — define before launching what "good" means (e.g., opted-in users publish ≥X% more often, or capture-completion ≥Y%). **Default: instrument first, set the bar with the product owner before evaluating.**
- **Timezone changes mid-day / travel** — which tz schedule applies if a user moves. **Default: bind to the tz at first prompt of the day; revisit if it causes confusion.**
- **Waking-hours window + count** — defaults 09:00–22:00 and 6; server-tunable. **Default: tune live during the test.**
- **Quiet/duplicate prompts** for users in multiple groups — N/A, because the moment is per-user/per-timezone (not per-group), so a user gets exactly one schedule regardless of group count.
- **Photo vs video per moment** — both allowed (same limits as M6). **Default: allow both; short videos auto-trimmed by the montage.**
